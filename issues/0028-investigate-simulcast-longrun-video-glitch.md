# サイマルキャストの長時間接続時に発生する映像の乱れを調査する

- Priority: Medium
- Created: 2026-06-03
- Completed:
- Model: Opus 4.8
- Branch:

## 目的

サイマルキャスト構成で長時間接続した際に不定期に発生する映像の乱れの原因を調査し、再現条件と発生メカニズムを明確にする。

## 優先度根拠

- 映像品質に直接影響する事象であり、利用者の体感に関わる。
- 一方で発生が不定期かつ発生契機が特定できておらず、まずは原因調査が必要な段階のため Medium とする。

## 現状

サイマルキャストでの相互送受信を長時間行うと、Android 側で不定期に映像の乱れが発生する。発生は不定期で、十数秒映像が乱れた後に正常へ戻る。既存アプリでも発生していたことから、以前から存在する事象と考えられる。

調査で得られている情報は以下のとおり。

- 受信側 Android の rtc_stats では、`inbound-rtp`（video）に `packetsLost` や `nackCount` の増加は見られず、受信側の通信は安定していた。
- 送信側の rtc_stats では `qualityLimitationDurations` の `"bandwidth"` に値が入っており、帯域不足が示されていた。
- Android の Logcat アプリログには明確な異常ログは出ていない。カメラ系のログ（環境光不足によるバンディング検出失敗など）は出ていたが、本事象との関連は不明。
- メモリ・CPU のグラフに明確な異常は見られなかった。
- 配信側ツールに表示された送受信映像には問題がなかった。

確認環境の概要は以下のとおり。

- クライアント: Android 端末（Pixel 系）でサイマルキャスト映像を相互送受信。libwebrtc は複数バージョンで確認。
- 対向クライアント: PC ブラウザ。
- コーデック: VP9 を指定。

## 設計方針

- 調査タスクとして扱い、まず原因の切り分けを行う。
- 送信側の `qualityLimitationDurations` の `"bandwidth"` が立つ事象に着目し、帯域推定・送信ビットレート制御（サイマルキャストの各レイヤーのビットレート配分）と映像の乱れの相関を確認する。
- 映像の乱れがエンコード側で起きているのか、ネットワーク帯域変動への追従で起きているのかを rtc_stats の時系列から切り分ける。
- 再現性を高めるため、ダミー映像送信など決まった品質の映像入力での再現を検討する（別 issue のダミー映像送信機能と関連）。

## 完了条件

- 映像の乱れが発生する条件（帯域・コーデック・レイヤー構成など）を特定し、発生メカニズムを説明できること。
- 修正が必要な場合は、対応方針を別 issue として切り出すか本 issue で方針を定めること。
