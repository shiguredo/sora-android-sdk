# 映像エフェクト機能を SDK に組み込む（または別パッケージで配布する）

- Priority: Low
- Created: 2026-06-03
- Completed:
- Model: Opus 4.8
- Branch:

## pending 理由

映像エフェクト機能で利用しているライブラリが 16 KB ページサイズに未対応であり、SDK への組み込みとするかドキュメント化に留めるかの設計判断が未決のため pending とする。

## 目的

現在はサンプルコードの一部として実装されている映像エフェクト機能を、SDK 本体へ組み込むか、または別パッケージとして配布できるようにブラッシュアップする。

利用者が映像フレームに対する加工処理を簡単に組み込めるようにすることが目的である。

## 優先度根拠

- 映像エフェクト機能はサンプルとして既に存在しており、SDK へ組み込まないと利用できないわけではない。
- 利用しているライブラリが 16 KB ページサイズサポートに未対応であり、前提となる依存ライブラリの整備が先行課題となるため緊急性は低く Low とする。

## 現状

- 映像エフェクト機能はサンプルコード側で実装されており、SDK 本体には含まれていない。
- 加工対象の映像フレームは GPU メモリ上に存在し、現状の処理は次の流れになっている。
  - カメラからのフレームが GPU メモリに確保される。
  - コールバックを受けてエフェクト処理が発動する。
  - フレームを CPU メモリに取得する（この時点で I420）。
  - I420 を RGBA に変換する（libyuv による変換で、SIMD 利用までに留まる）。
  - RGBA を GPU メモリに書き込み、GPU 側でエフェクト処理を実行する。
  - 処理結果の RGBA を CPU メモリにコピーする。
  - RGBA を I420 に変換し直す。
  - I420 を `VideoFrame` で包んで libwebrtc に返す。

## 設計方針

- GPU メモリと CPU メモリ間の往復を減らせないか調査する。加工ライブラリが GPU 上のデータに直接適用できれば、C / JNI のコードや変換処理を削減できる可能性がある。
- libwebrtc が提供する libyuv 相当の API（`YuvConverter` / `YuvHelper` 等）が利用できるか調査する。
- 依存ライブラリが増えるため、SDK のコア部分と分離し、必要な利用者だけが依存に追加できる別モジュール構成を検討する。
- クラス名の接頭辞が libwebrtc の API と誤解を招くため、接頭辞を見直す。
- 利用しているライブラリが 16 KB ページサイズサポートに対応していることを前提とする。
- SDK へ組み込むか、利用方法をドキュメント化するに留めるかを判断する。

## 完了条件

- 映像エフェクト機能を SDK 本体へ組み込むか、別パッケージで配布するか、ドキュメント化に留めるかの方針を決定すること。
- 採用方針に応じて、依存ライブラリの 16 KB ページサイズ対応状況を確認すること。
